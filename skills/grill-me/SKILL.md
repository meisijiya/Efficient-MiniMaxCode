# Grill Me — 需求深挖流程

> 来自 addy 的 `interview-me` + matt 的 `grill-me` 融合。**一次一个问题，反复挖到 95% 置信度**才进 spec。

## 触发场景

- `spec-miner` agent 启动后立即 load
- 任何"用户给了模糊需求"的场景
- 用户说"差不多就行"时**更要 grill**（"差不多"=用户没参与）

## 核心原则

> **不替用户说话。一次一个问题。挖到 95% 才停。**

| 原则 | 含义 |
|------|------|
| **一次一个** | 永远不要一次问多个问题 |
| **开放性** | 第一问必须是开放的（不能 yes/no） |
| **不假设** | 不替用户说"我猜你想要 X"——问 |
| **不替代** | "差不多" = 你定 = 用户没参与。逼出精确 |
| **记事实** | 不仅记"用户提到 XX"，记"用户说 XX 是 X" |

## 完整对话流程（spec-miner 用）

### Round 1: 第一问（开放性）

**模板**：
- 新功能：`"你想做 [XX]，**主要痛点**是什么？现在怎么解决？"`
- 修改：`"[XX] **现在**什么样？哪里让你觉得不行？"`
- 性能：`"你说'慢'——**慢到什么程度**？现在 [时长]？你想 [时长]？"`
- Bug：`"[XX] **什么时候**出现的？必现还是偶现？影响多少人？"`
- 探索：`"你想做 [XX]——**为什么**？如果不做会怎样？"`

**禁止**：
- ❌ "你想要 A 还是 B？"（强迫二选一）
- ❌ "你是想要 A/B/C 哪个？"（3 选也不够）
- ❌ "你能接受 X 吗？"（yes/no 不是 grill）
- ❌ 一次问 5 个问题

### Round 2-4: 用户流（最常缺）

**模板**：
- "用户**怎么开始**这个流程？从哪个页面 / 哪个按钮？"
- "**进行中**会看到什么？需要输入什么？"
- "**成功**长什么样？成功时用户看到什么？"
- "**失败**长什么样？失败时用户看到什么？"
- "**结束**后用户去哪？"

**注意**：
- 失败 ≠ 报错（失败是"业务上的失败"，比如支付被拒、订单已存在）
- 不只问 happy path

### Round 5-6: 边界（最常模糊）

**模板**：
- "[XX] 算**成功**还是**失败**？边界在哪？"
- "**并发**多少？QPS / TPS 量级？"
- "**数据量**多少？列表最多几条？上传最大几 MB？"
- "**超时**多久算失败？"
- "**重试**吗？重试几次？"
- "**幂等**吗？同一请求两次会怎样？"

### Round 7-8: 非目标（最常被忽略）

**模板**：
- "这个功能**不做**什么？"
- "如果只能保留 1 个 sub-feature，砍哪个？"
- "什么场景**明确不在**这次范围？"
- "**v2 才做**的能想到吗？"

**为什么非要问**：用户不说 = 你猜 = 必然错。**必须显式问**"不做 X"。

### Round 9-10: 验收标准（最常漏）

**模板**：
- "怎么判断这个功能**做完了**？"
- "**用户能测出来**什么 = 成功？"
- "**自动化测试**能跑通什么？"
- "**产品验收**时 PM 怎么判断过 / 不过？"

**格式**：每条验收标准 = 1 个可测条件（不是"功能完善"这种空话）

### Round 11+: 依赖 / 约束

**模板**：
- "依赖什么**外部服务**？API / DB / 中间件？"
- "必须用 / 不能用**什么技术**？"
- "**性能 / 安全 / 合规**有要求吗？"
- "**时间**限制？必须在 [日期] 前？"

## 实战对话示例

```
[Round 1]
你: 你想做 SSO 登录 —— 主要痛点是什么？团队现在登录怎么做的，哪里不行？
用户: 我们现在用账号密码登录，员工老忘密码，admin 重置密码每天 50 次。
你: [记下] 痛点 = admin 重置 50 次/天。

[Round 2]
你: 你想用哪个 SSO provider？Google / Microsoft / GitHub / 自建 SAML？
用户: 都可以，你觉得呢？
你: ⚠️ 不要"我觉得"！这是用户问题。
   你: 你的用户是技术人员还是普通员工？技术栈偏 Google Workspace 还是 Microsoft 365？
用户: 我们的客户公司大部分用 Microsoft 365。
你: [记下] Microsoft 365 / Azure AD 是首选。

[Round 3]
你: SSO 登录失败时，回退到账号密码登录还是直接报错？
用户: 回退账号密码吧。
你: [记下] 失败回退 = 账号密码。

[Round 4]
你: 用户怎么开始 SSO 登录？从登录页选"用 Microsoft 登录"按钮？
用户: 对。

[Round 5]
你: 这个流程的成功标准是什么？给一个具体可测的。
用户: 公司域名邮箱登录时，跳到 Microsoft，登录成功回到我们网站。
你: [记下] AC: <公司域名> 邮箱 → 跳 Microsoft → 登录成功 → 回到我们网站。

[Round 6]
你: 如果用户用 Gmail 登录呢？也算 SSO 还是走账号密码？
用户: 走账号密码。
你: [记下] 非 Microsoft 域名 → 账号密码登录。

[Round 7]
你: 这个功能**不做**什么？比如说不做多租户？或者不做 SCIM？
用户: SCIM 不要，太复杂。多租户需要。
你: [记下] 不做 SCIM / 做多租户。

[8-10 rounds 后]

[Round N]
你: 自检一下 —— 我列下我以为我知道的，你看看对不对：
   - 痛点：admin 重置 50 次/天
   - Provider: Microsoft 365 / Azure AD
   - 失败回退: 账号密码
   - 成功标准: 域名邮箱 → 跳 Microsoft → 回我们网站
   - 不做 SCIM、做多租户
   哪里不对 / 漏了？
用户: 对的。
你: [进入 spec 输出阶段]
```

## 95% 置信度自检清单

每轮 grill 后 / 写 spec 前必查：

- [ ] **用户流**（开始 / 进行 / 失败 / 结束）—— 4 步都清楚？
- [ ] **边界**（什么算成功 / 失败 / 并发 / 数据量 / 超时）—— 至少 3 项清楚？
- [ ] **非目标**（明确不做）—— 至少 1 项明确？
- [ ] **验收标准**（怎么测 = 成功）—— 至少 3 条可测？
- [ ] **依赖 / 约束**（外部服务 / 技术栈 / 时间）—— 至少 1 项明确？
- [ ] **任何"差不多 / 应该 / 大概"模糊点都问过了吗**？—— 0 个？

**5/6 通过 = 进 spec**
**4/6 = 继续 grill**
**3/6 以下 = 重新挖（用户没说清楚）**

## Anti-rationalization 完整版

| 借口 | 反驳 | 怎么办 |
|------|------|--------|
| "用户已经说清楚了吧" | 90% 只说 60% | 再问 3 个问题 |
| "我先猜一个，用户会改" | 你猜 = 你引入歧义 | 问，不猜 |
| "问太多用户烦" | 5-10 轮比 3 轮改来改去省时间 | 继续 |
| "细节实现不用管" | 验收 / 边界 / 非目标不是"实现细节" | 必问 |
| "我可以假设" | 假设 = 失败源 | 每个假设问一遍 |
| "差不多就行" | "差不多" = 你定 = 用户没参与 | 逼出精确 |
| "用户没时间 grill" | 5 轮 × 1 分钟 = 5 分钟 < 改 3 轮 × 30 分钟 | 继续 |
| "我擅长猜用户意图" | 自圆其说 = 隐藏失败 | 停止猜测 |
| "这个简单不需要 grill" | 简单需求最常错 | 也走 3-5 轮 |
| "用户要 XX 我就做 XX" | 偷懒借口 | 用户要 X ≠ X 是对的解 |

## 红线

- ❌ 一次问多个问题
- ❌ 替用户做选择（"你想要 A 还是 B" 强迫二选一）
- ❌ 跳过非目标（必问"明确不做"）
- ❌ 跳过验收标准（必问"怎么测"）
- ❌ 跳过边界（必问"什么算失败"）
- ❌ 在 95% 置信度前停
- ❌ 用户说"差不多"就停
- ❌ 自己猜用户意图

## 与 spec-miner agent 的关系

- spec-miner agent.md 提供**骨架**（阶段、4 原则、Anti-rationalization 摘要）
- **本 skill 提供详细**（对话模板、自检清单、5/6 评分）
- agent 启动后**必须 load 本 skill** 才能完整 grill

## 与 mavis 的关系

如果 grill 8 轮还没到 95%：
- **不要无限继续**——> 8 轮通常意味着用户说不清楚
- 让 mavis 决定"放弃 / 重写需求 / 拆小"

## 一句话总结

> **你是记者不是预言家。一次一个问题。挖到 95% 才动笔。**
