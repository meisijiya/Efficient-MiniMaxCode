# Installation Guide

> **Where does every file go, and how does Mavis read it?**

This document is the **definitive reference** for installing, updating, and uninstalling Efficient-MiniMaxCode. If you ever wonder "is this file even being read?", this is the place.

---

## TL;DR

```powershell
# One-line install (Windows PowerShell)
git clone https://github.com/meisijiya/Efficient-MiniMaxCode.git
cd Efficient-MiniMaxCode
.\install.ps1

# Or macOS / Linux
git clone https://github.com/meisijiya/Efficient-MiniMaxCode.git
cd Efficient-MiniMaxCode
chmod +x install.sh && ./install.sh
```

**That's it.** Mavis will pick up the new agents / skills the next time it starts (or after `mavis agent list` is run).

---

## How Mavis Loads Things

Mavis Code reads from a **single root directory** (per-user):

| Platform | Path |
|----------|------|
| Windows | `%USERPROFILE%\.mavis\` (e.g. `C:\Users\<you>\.mavis\`) |
| macOS / Linux | `~/.mavis/` |

This document calls it `~/.mavis/` for brevity. **Substitute your real path** if you're on Windows.

The two key subdirectories:

```
~/.mavis/
├── agents/
│   └── <agent-name>/
│       └── agent.md          ← User-controlled content
└── skills/
    └── <skill-name>/
        └── SKILL.md          ← User-controlled content
```

Everything else under `~/.mavis/` is either:
- **Auto-generated by the Mavis daemon** (don't manage manually)
- **Runtime data** (sessions / scratchpads / uploads)
- **Internal state** (config.yaml, daemon database)

## The Full File Map (Every File & Where It Lives)

### Files YOU Manage (these go in the repo)

| File | Where it goes in `~/.mavis/` | What it does |
|------|------------------------------|--------------|
| `agents/<name>/agent.md` | `~/.mavis/agents/<name>/agent.md` | **User prompt overlay** — appended to the built-in prompt at runtime |
| `skills/<name>/SKILL.md` | `~/.mavis/skills/<name>/SKILL.md` | **Skill definition** — auto-loaded by description match |

That's it. **Only 2 file types** in the repo. Everything else is daemon-generated.

### Files DAEMON Generates (do NOT put in the repo)

| File | Where it lives | Generated when | Purpose |
|------|----------------|----------------|---------|
| `agents/<name>/PERSONA.md` | `~/.mavis/agents/<name>/PERSONA.md` | On `mavis agent new` | Mirror of the `--persona` argument (for display) |
| `agents/<name>/config.yaml` | `~/.mavis/agents/<name>/config.yaml` | On `mavis agent new` | Per-agent config stub (currently empty `{}`) |
| `agents/<name>/.builtin-prompt-layout-v2` | `~/.mavis/agents/<name>/.builtin-prompt-layout-v2` | On `mavis agent new` (built-in agents only) | Marks that the agent uses a built-in prompt layout (version 3) |
| `skills/<name>/_meta.json` | `~/.mavis/skills/<name>/_meta.json` | On first skill load | Skill metadata (id, version, updated_at, platform) |

**Why exclude them**:
- They are **re-creatable** by the daemon on first run
- They contain **machine-specific data** (timestamps, internal IDs)
- Tracking them in git causes **merge conflicts** every reinstall
- They are **not the source of truth** — the daemon reads `agent.md` and `SKILL.md`, not these

### Runtime / State (do NOT touch)

| Path | Purpose | Re-creatable? |
|------|---------|---------------|
| `~/.mavis/agents/<name>/workspace/` | Per-agent working directory | Yes (empty after first start) |
| `~/.mavis/agents/<name>/scratchpads/` | Agent scratch notes | Yes |
| `~/.mavis/agents/<agent>/.builtin-skills/<skill>/` | Mavis-specific built-in skills (e.g. `mavis-team`, `mavis-doctor`) | Yes (only for `mavis` agent) |
| `~/.mavis/sessions/` | Session history | No (deletable but loses history) |
| `~/.mavis/scratchpads/` | Cross-session scratch notes | No |
| `~/.mavis/plans/` | Team plan work directory | Yes (plans regenerate) |
| `~/.mavis/uploads/` | User-uploaded files | No (user data) |
| `~/.mavis/memory/` | Memory system (user / agent / project) | No (deletable but loses learning) |
| `~/.mavis/credentials/` | API keys / tokens | No (regenerable from provider) |
| `~/.mavis/config.yaml` | Daemon config (LLM providers, model whitelist) | No (loses provider config) |
| `~/.mavis/session.db` | Daemon database (sessions, agents registry) | No (loses session list) |

## Install — Detailed Steps

### Step 1: Clone the repo

```bash
git clone https://github.com/meisijiya/Efficient-MiniMaxCode.git
cd Efficient-MiniMaxCode
```

### Step 2: Run the installer

**Windows (PowerShell)**:
```powershell
.\install.ps1
```

**macOS / Linux (bash)**:
```bash
chmod +x install.sh
./install.sh
```

The installer will:
1. Detect your `~/.mavis/` directory
2. Copy `agents/<name>/agent.md` → `~/.mavis/agents/<name>/agent.md`
3. Copy `skills/<name>/SKILL.md` → `~/.mavis/skills/<name>/SKILL.md`
4. Skip any agent / skill that is daemon-managed (built-in agents)
5. Run `mavis agent list` to trigger re-registration

**Safe by default**:
- Does NOT overwrite the daemon-generated files (PERSONA.md, config.yaml, _meta.json)
- Does NOT delete anything
- Reports what was changed

### Step 3: Verify

```powershell
.\verify.ps1
```

You should see all 13 agents listed in `mavis agent list` and all 23 skills loaded.

### Step 4: Restart Mavis Code (optional)

If Mavis Code is already running, restart it to pick up the new content. If it's not running, just start it normally.

## What the Installer Skips (And Why)

The install script **skips** the following files (they're in the repo for **documentation only**, not for installation):

| Path in repo | Why skipped |
|---------------|-------------|
| `agents/<name>/config.yaml` | Daemon-generated, always empty `{}` |
| `agents/<name>/PERSONA.md` | Daemon-generated mirror of `--persona` |
| `agents/<name>/.builtin-prompt-layout-v2` | Daemon-generated marker (built-in agents only) |
| `skills/<name>/_meta.json` | Daemon-generated metadata |

If you really want to copy them anyway (e.g. to preserve timestamps), use `install.ps1 -ForceAll`. **Not recommended** — these files contain machine-specific IDs.

## Update Workflow

When the repo gets new agents / skills (or updates to existing ones):

```bash
git pull origin main
./install.ps1   # or .\install.ps1 on Windows
./verify.ps1
```

The installer will:
- Add new agents / skills that don't exist locally
- Update existing `agent.md` / `SKILL.md` (the user-controlled files)
- Leave daemon-generated files alone

**No merge conflicts** because we only track the source-of-truth files.

---

## Uninstall

```powershell
.\uninstall.ps1
# or
./uninstall.sh
```

The uninstaller will:
1. Ask for confirmation (unless `-Force` is passed)
2. Move all `agent.md` files installed by this repo to `~/.mavis/.efficient-backup-<timestamp>/`
3. Move all `SKILL.md` files installed by this repo to the same backup
4. Remove the files from `~/.mavis/agents/<name>/` and `~/.mavis/skills/<name>/`
5. Keep daemon-generated files (config.yaml, PERSONA.md, _meta.json) — those are the daemon's responsibility

**To restore**, copy files back from the backup directory.

## Troubleshooting

### "I copied the files but `mavis agent list` doesn't show new agents"

1. Make sure you copied to the **correct** `~/.mavis/` path
2. Restart Mavis Code (or `mavis daemon restart`)
3. Check the file structure: `<name>/agent.md` (not `agent.md` directly in agents/)

### "Agent shows up but ignores my agent.md"

The file must:
- Be exactly at `~/.mavis/agents/<agent-name>/agent.md`
- Start with the marker comment: `<!-- mavis:builtin-agent-md-stub v2 -->`
- Be valid UTF-8 (not UTF-8 with BOM)

### "Skill doesn't auto-load"

1. Check the file is at `~/.mavis/skills/<skill-name>/SKILL.md`
2. Check the YAML frontmatter at the top of SKILL.md:
   ```yaml
   ---
   name: my-skill
   description: "Trigger keywords here"
   ---
   ```
3. The `description` field is what Mavis matches — make sure it includes relevant keywords

### "I see extra files in the repo I didn't expect"

That's intentional. The repo includes:
- `agents/<name>/config.yaml` and `PERSONA.md` and `.builtin-prompt-layout-v2` — **for reference**, showing what the daemon generates
- `skills/<name>/_meta.json` — same reason
- These are **not copied** by the installer

## Why This Split Matters

**The fundamental principle**:
> **Source of truth (in git) ≠ machine state (on disk)**

- `agent.md` and `SKILL.md` are **the actual prompts** the LLM reads.
- Everything else under `~/.mavis/` is **scaffolding** the daemon needs to function.

If you put the scaffolding in git:
- You get **merge conflicts** every reinstall (different IDs, different timestamps)
- You **confuse new contributors** ("do I need to maintain this empty `{}`?")
- You **lose portability** (the IDs and timestamps are machine-specific)

If you put only the source-of-truth files in git:
- **Clean reinstalls** forever
- **No merge conflicts**
- **The repo documents what matters** (the prompts), not the noise (the metadata)

This is why the install script copies only `agent.md` and `SKILL.md`.
